Question 1

Different groupId's

<project>
  <artifactId>myproject</artifactId>
  ...
  <dependencies>
    <dependency>
      <groupId>javassist</groupId>
      <artifactId>javassist</artifactId>
      <version>3.12.1.GA</version>
    </dependency>
    <dependency>
      <groupId>org.javassist</groupId>
      <artifactId>javassist</artifactId>
      <version>3.16.1-GA</version>
    </dependency>
  </dependencies>
</project>

Different groupId's

Which jars on the classpath?

  • javassist-3.12.1.GA.jar
  • javassist-3.16.1-GA.jar
  • javassist-3.12.1.GA.jar
    + javassist-3.16.1-GA.jar
  • neither
  • Maven error
  • I don't know

Answer different groupId's

Which jars on the classpath?

  • javassist-3.12.1.GA.jar
  • javassist-3.16.1-GA.jar
  • javassist-3.12.1.GA.jar
    + javassist-3.16.1-GA.jar
    (+3 points)
  • neither
  • Maven error
  • I don't know
    (+1 point)

Solution different groupId's

  • Don't change your groupId's or artifactId's
  • If you do change them,
    relocate them in the repository too 
    • <project>
  <modelVersion>4.0.0</modelVersion>
  <groupId>javassist</groupId>
  <artifactId>javassist</artifactId>
  <version>3.16.1-GA</version>
  <distributionManagement>
    <relocation>
      <groupId>org.javassist</groupId>
    </relocation>
  </distributionManagement>
</project>
    • http://maven.apache.org/guides/mini/guide-relocation.html

meanwhile in Maven Central ...

groupId's of Sun's JAXB implementation

Workaround different groupId's

Exclude wrong groupId's and artifactId's

<dependencies>
  <dependency>
    ...
    <exclusions>
      <exclusion>
        <!-- javax.xml is wrong, com.sun.xml.bind is correct -->
        <groupId>javax.xml</groupId>
        <artifactId>jaxb-impl</artifactId>
      </exclusion>
    </exclusions>
  </dependency>
  <dependency>
    <groupId>com.sun.xml.bind</groupId>
    <artifactId>jaxb-impl</artifactId>
    <version>2.2.6</version>
  </dependency>
</dependencies>

When in doubt, prefer the GAV with

  • the highest version
  • the most recent versions

Workaround: Fail fast with enforcer

Enforce that every (transitive) dependency is resolved to it's specified version or higher.

<plugin>
  <artifactId>maven-enforcer-plugin</artifactId>
  <executions>
    <execution>
      <goals><goal>enforce</goal></goals>
      <configuration>
        <rules>
          <requireUpperBoundDeps/>
        </rules>
      </configuration>
    </execution>
  </executions>
</plugin>

Later releases (5.3, 5.4, 5.5, ...)
no longer use M in version numbering.

Workaround conflicting cousins

  • Use lexicographically friendly version numbering
    • Tokenize on dots (.), then dashes (-)
    • Letters are smaller than numbers: a < 1
    • Sequential ciphers from a number: 2 < 10
    • 1.0.0.Beta3 < 1.0.0.Beta11 < 1.0.0.CR2
      < 1.0.0.Final < 1.0.1.Beta1
  • For example: JBoss version guidelines
    • major.minor.micro.Alpha[n]
    • major.minor.micro.Beta[n]
    • major.minor.micro.CR[n]
    • major.minor.micro.Final
    • https://community.jboss.org/wiki/JBossProjectVersioning
  • Use enforcer rule requireUpperBoundDeps
    • Compares versions lexicographically

1 thing I don't like about Maven

  • "Many projects do not follow lexicographically versioning."
  • When Maven detects a conflict,
    it just picks the nearest
    (= highest level, first index)
  • "Comparing financial stocks is hard
    so I'll just buy the first one in the list."

Conflict resolution in practice ...

  • Most popular projects are backwards compatible
  • 2 conflicting jars in 1 classpath = trouble

Common sense solution:
Retain the highest version
(lexicographic comparison)

  • So why doesn't maven do this?

Shading

Solution shading

Don't depend on shaded jars.

<project>
  <artifactId>myproject</artifactId>
  ...
  <dependencies>
    <dependency>
      <groupId>org.jboss.weld.se</groupId>
      <-- <artifactId>weld-se</artifactId> is evil -->
      <artifactId>weld-se-core</artifactId>
      <version>1.1.8.Final</version>
    </dependency>
  </dependencies>
</project>
